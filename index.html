⚙️Pipeline Steps Overview

The pipeline preserves the original pdftocairo SVG structure — it does not rebuild SVGs from scratch. XML tree walking, not regex, is used throughout.

1
Pre-process EPS → PDF → raw SVG

Fix BoundingBox atend bugs, clean duplicate xmlns attributes. Convert: EPS → PDF (Ghostscript with -dEPSCrop) → raw SVG (pdftocairo). Keep the PDF — it's used for PNG rendering.

1d
Clip mask preprocessing

Strips manufacturing junk by finding the sign border rectangle and removing everything outside it — profile views, dimension callouts, title blocks. Runs AFTER EPS→PDF→SVG, BEFORE face detection.

2
Parse raw SVG and detect the sign face

Use the nesting approach: find all large filled rectangles, pick the one containing the most text elements inside its bounds. Never pick the largest dark path — that selects the assembly view, not the sign face.

3
Classify and filter elements

Keep: face backer, backer paths (FOH), clip groups overlapping the face, decorative paths. Extract position from text in magenta boxes (variable fields). Remove everything else: title blocks, dimension lines, blue guide lines, pink production borders, magenta dashed boxes.

4
Extract text positions and compute font sizes

Cluster glyphs by Y position. Read true baselines from raw SVG <use y> — never from pdfplumber. Compute font size using BoundsPen actual glyph bounds, not advance width.

5
Rebuild defs and content group

For variable signs: remove glyph symbols from production SVG. Always skip <defs> when pruning — walking into defs deletes glyph paths and breaks all text rendering. Use string insertion (never ElementTree) when modifying SVGs.

6
Live text and font embedding

Base64-encode font files and embed in <style> @font-face. Add live <text> elements at extracted baseline positions. Placeholder text must be ALL CAPS matching the art file.

7
FIELD_ANCHORS — bounding boxes

Width priority: magenta box width → glyph width + 10% → 3/8" guard rail. Height = cap height + 1.5pt top padding (never font_size). Box bottom sits exactly on the text baseline. Every field needs anchor_point, anchor_h, and anchor_v.

8
ViewBox and sizing

Tight viewBox enclosing all kept visual elements. FOH signs: include the backer below the face. SVG root width/height set in inches at 1:1 physical scale. Add xmlns:xlink if missing.

9
Generate PNG preview

Render from intermediate PDF via Ghostscript (not cairosvg — it can't load base64 fonts). Crop to sign face bounds. Remove magenta with dilated mask filled in face background color — never white.

10
Generate _fields.json

Write sign metadata, face bounds, and full fields array. Every text/braille field must include bbox, anchor_point, anchor_h, anchor_v. Braille fields must include braille_spacing.gap_pt (ADA requirement).

13
Quality checklist and batch audit

Run per-SKU smoke tests and the batch audit script. Check file counts (5 for variable, 4 for static), xlink:href survival in preview SVGs, empty symbol detection, font audit, and AI file cross-reference. Never skip.

🚨 Three-Point Approval Criteria — a sign is approved only when: (1) braille spacing is 1:1 to EPS measurement, (2) SVG is at 1:1 physical scale, and (3) every field has an explicit anchor_point. All three must pass.

Key Bounding Box Rules

RuleValueWhy It Matters
Minimum edge clearance 3/8" (27pt) Fabrication requirement — no bounding box may extend closer than 27pt to any sign face edge
Box height formula cap_height + 1.5pt Raw cap height alone clips X/M/A glyphs. Box bottom sits exactly on the text baseline.
Width source priority Magenta box → Glyph width + 10% → Guard rail The designer already sized the magenta box for the widest expected content. Trust it.
Anchor point required Every text and braille field Without an anchor, the renderer guesses alignment. Short and long values won't align correctly.
Braille spacing braille_spacing.gap_pt ADA/fire inspection requirement. If missing or wrong, the sign fails inspection.
Separate box per line One bbox per text line Multi-line signs: each line gets its own box at its own baseline. Never merge lines.

Static Signs — 4 files required

Static signs (fire evacuation, accessibility symbols, decals, COA letters). fields[] is empty.

FileNaming PatternPurposeRequired
Vector SVG {SKU}-vector.svg Production file — original glyph paths preserved intact. No live text, no FIELD_ANCHOR rects. Required

💡 Naming convention matters. SVG and PNG files use hyphens: 88622-vector.svg. JSON and HTML files use underscores: 88622_fields.json. The developer's import script expects this exact pattern — wrong names cause silent failures.

🔧Common Troubleshooting Issues with Claude

These are real failures that have occurred in production — each one has a known cause and fix. When Claude's output looks wrong, check here first before escalating.

Visual Bug SVG renders blank — no text visible

Root cause: Python's ElementTree renamed the xlink namespace to ns1: when writing the SVG. Every <use xlink:href="#glyph…"> became <use ns1:href="…">, which browsers refuse to resolve.

How to confirm: Open the SVG in a text editor and search for xlink:href. If it reads ns1:href, this is the issue.

Fix: SVGs must never be round-tripped through ElementTree after initial construction. All modifications to preview-vector and REVIEW SVGs must use string insertion:

svg = open(path).read()
overlay = '<rect x="..." y="..." .../>'
out = svg.replace("</svg>", overlay + "\n</svg>")
assert 'xlink:href' in out, "ET namespace bug"

Prevention: Claude runs a smoke test (assert 'xlink:href' in text) after writing every SVG. If this assert fires, the file cannot be delivered.

Visual Bug All text renders blank after preview SVG was modified

Root cause: When pruning the raw SVG, the prune walker descended into <defs> and deleted the glyph <symbol> paths. Every <use> reference then resolves to an empty symbol and renders nothing.

How to confirm: Count <symbol> elements in the SVG — if any are empty (no child path), the defs were walked.

Fix: Any prune walker must skip <defs> entirely:

if tag_local == 'defs':
    continue  # NEVER walk into defs

After pruning, assert no empty symbols remain.

Alignment Bug Bounding boxes don't sit on the text — gap below text or box clips the top

Root cause A — Box too tall: bbox_h was set to font_size instead of cap_height + 1.5pt. Font size includes empty space above cap height and below baseline — the box appears to float above the text with empty space below.

Root cause B — Baseline wrong: Baseline was read from pdfplumber's char["bottom"], which sits ~0.282 × font_size below the true baseline. This shifts every box down by that amount.

Fix: Read baselines from raw SVG <use y> values only. Compute height as:

cap_pt = (sCapHeight / upm) * font_size
bbox_h = cap_pt + 1.5   # 1.5pt top pad
bbox_y = baseline_y - bbox_h  # box bottom = baseline
Visual Bug Magenta dots or ghost outlines still visible in the PNG

Root cause: Naive pixel replacement (arr[pink_mask] = [255,255,255]) misses anti-aliased dash edges and fills with pure white instead of the face background color — leaving a visible halo and color mismatch.

Fix: Use a dilated mask and fill with the brand face background color:

# Dilate 2px to swallow anti-aliased edges
dilated = mask_img.filter(ImageFilter.MaxFilter(5))
# Fill with FACE COLOR, not white
arr[dilated > 0] = brand['face_bg_rgb']

Face background color values are in the brand technical reference file. Never use [255,255,255].

Scale Bug All measurements are wrong on a multi-face EPS file

Root cause: When a designer puts two face variants side by side on one EPS sheet, the sign scales down to fit. The raw SVG coordinates are at 18pt/inch instead of the standard 27pt/inch. All cap heights, margins, and font sizes calculated at 27pt/inch will be wrong.

How to confirm: Measure the face width in raw SVG pt, divide by the physical sign width in inches from the MSDQ. If the result is ~18, not ~27, the file is scaled down.

Fix: Always cross-check scale before any measurement:

pts_per_inch = face_w_raw / phys_w_inches
# 18 pt/in = two-face sheet
# 27 pt/in = standard single-face

All margin, font, and cap height calculations must use the derived pts_per_inch. Physical font size in pt will match the brand spec regardless of EPS scale.

Pipeline Error Claude processed a file that should have been skipped

Certain files appear in every brand folder and must never be processed. If Claude includes them in the batch output, delete the output and flag the session.

  • CREDIT.eps — credit/attribution sheets
  • INSP.eps — inspection protocol sheets
  • INSTALLATION.eps — installation instruction sheets
  • LINK LAYOUT*.eps — plan layout diagrams (any variant)
  • Any EPS whose filename does not start with a numeric SKU
  • Installation photos or 3D renders of signs mounted on walls
  • Files from a different brand that ended up in the same folder

These are logged in the batch summary. If the skip log is missing, ask Claude to recheck before delivery.

Face Detection Claude selected the wrong face (assembly view or side view)

Root cause: The largest dark path in the EPS is often the assembly or perspective view, not the sign face. Picking by size alone selects the wrong view.

Correct approach: The pipeline uses the nesting method — finds all large filled rectangles and picks the one containing the most text elements within its bounds. This reliably identifies the face regardless of draw order.

If this still fails: Ask Claude to filter by expected dimensions from the brand reference (physical sign width × pts_per_inch). Then pick from the top row of the grid (lowest Y coordinate) for the 1-line placeholder configuration.

Escalate to Elaine when: Multiple candidate faces exist and Claude cannot determine which is correct, or when blue guide lines cannot be removed by color filtering (request the clean .ai source file instead).

Output Error FIELD_ANCHORS_Review.html is missing for a variable sign

The HTML review file is mandatory for every variable sign — no exceptions, even if there is no braille. This is the file that lets you and the developer verify field placement before integration.

If the file is missing, tell Claude: "Generate the FIELD_ANCHORS_Review.html for SKU [number]. It is missing from the output folder."

The HTML must contain: the raster PNG (base64 embedded), one spec card per field showing font/size/bbox/anchor/placeholder, an ADA braille section if applicable, and the full JSON dump at the bottom.

Font Error Wrong font name in fields.json

Font family names must exactly match the PostScript/family name of the installed font file. Incorrect names cause the developer's renderer to fall back to a system font, breaking the live preview.

Known wrong names that have appeared in production:

  • NewsGothicComMed → correct: NewsGothicCom-Medium
  • FuturaBT-Bold on a text field → correct: check brand reference
  • FuturaBT-MediumCondensed on a type: "text" field → this font is for braille only

Claude runs a font audit as part of Step 14. If the audit reports errors, correct the fields.json before delivery — never ship with font warnings.

Audit Error Batch audit passes but FIELD_ANCHORS files are missing

The original Step 13 audit did not check for HTML files. The Step 15 audit does. Always run Step 15 as the final check — it distinguishes live-text signs (5 files required) from static signs (4 files required) and checks for the HTML file on live-text SKUs.

Ask Claude: "Run the Step 15 file completeness audit on this batch." All SKUs must show a pass before handoff.

When to Stop and Escalate to Elaine

  • A sign has more or fewer magenta boxes than the dictionary specifies
  • Face detection found multiple candidates and cannot determine which is correct
  • Blue guide lines cannot be removed by color filtering — request clean .ai source file
  • Font files are missing — Claude will state which ones are needed
  • A sign appears to need splitting but no new SKU has been assigned yet
  • The EPS has a non-standard scale (two faces side by side)
  • A sign variant has different static text content — each variant needs its own SKU

🤖When to Use Sonnet vs Opus

Choosing the right model saves time and budget. The default for this pipeline is Sonnet. Switch to Opus only when deeper reasoning is genuinely needed.

Claude Sonnet
Default — use this most of the time
  • Static sign batches — fire evacuation, restrooms, decals, COA letters. No judgment needed; fast throughput matters.
  • Predictable variable signs — room numbers with one magenta box, single-field signs, brand name plates. Layout is unambiguous.
  • Running the audit scripts — Step 13 / Step 15 batch checks, font audit, AI file cross-reference.
  • Generating AI production files — straightforward cairosvg conversion loop.
  • Reading a brand MSDQ PDF and extracting sign specs, colorway names, font names.
  • Writing fields.json when field positions are already determined and confirmed.
  • Quick Q&A — "What's the bbox formula?" or "How do I name this file?"
Claude Opus
Use when judgment & depth matter
  • Building the sign dictionary for a new brand — classifying static vs variable, identifying colorways, determining SKU splits from a complex MSDQ.
  • Multi-face EPS split decisions — determining whether variants share one SKU or need separate ones. High-stakes — wrong calls require rework and workbook corrections.
  • Ambiguous face detection — when multiple large rectangles exist and the nesting approach is inconclusive.
  • Complex directional signs with 10+ fields and non-standard arrow-to-destination ratios.
  • New brand onboarding — building the technical reference from scratch, calibrating pts_per_inch, verifying fonts.
  • Signs flagged for Cowork review — braille zone placement judgment, stairwell informational signs, multi-subtype EPS files.
✅ Rule of thumb: Start with Sonnet. If Claude is uncertain, asking clarifying questions repeatedly, or producing outputs that need more than one correction, switch to Opus for that sign or task. You can switch mid-session without losing context on Cowork.

💬When to Use Chat vs Cowork

Chat and Cowork are two different ways of working with Claude. Understanding which to use for which task keeps the pipeline running without tool errors or context confusion.

Claude Chat
claude.ai — web or desktop, no file access
  • Questions and explanations — "What does anchor_h do?" or "Explain the difference between FOH and BOH."
  • Document review when you paste or upload content directly in the conversation — reading a MSDQ PDF you've uploaded, reviewing a fields.json you paste in.
  • Drafting content — writing sign descriptions for the workbook, composing a handoff email.
  • Troubleshooting diagnoses — paste in a fields.json or SVG snippet and ask what's wrong.
  • Quick one-off checks — "Is this font name correct for AmericInn?" or "Does this bbox formula look right?"
  • Learning and onboarding — asking conceptual questions about how the pipeline works before your first session.
Cowork
Claude desktop app — file system access, tools, skills
  • Running the pipeline — any task that involves reading EPS files, writing SVGs, running Python scripts, or producing output files.
  • Batch processing — processing an entire brand folder or subset of SKUs in one session.
  • Building sign dictionaries — reading the MSDQ PDF from the file system, writing the SKILL.md output.
  • Generating AI production files — cairosvg loop needs access to the file system.
  • Running audit scripts — Step 13 / Step 15 checks require reading the output folder.
  • Signs requiring Elaine's judgment — Cowork sessions are where Elaine reviews output visually and approves SKUs before handoff.
  • Anything that writes, reads, or moves files on the processing machine.
⚠️ Don't try to run the pipeline in Chat. Chat Claude has no access to your file system — it can't read EPS files, write SVG outputs, or run Python. Attempting to run pipeline tasks in Chat will result in Claude generating placeholder code it can't actually execute. Use Cowork for any task involving files.
✓ Chat is right for
  • Questions about pipeline concepts
  • Reviewing documents you paste in
  • Getting explanations of errors
  • Drafting workbook descriptions
  • Learning before your first Cowork session
✓ Cowork is right for
  • Any EPS → SVG conversion work
  • Reading or writing files on your machine
  • Running Python pipeline scripts
  • Batch processing and audits
  • Elaine's visual approval sessions

Pipeline Core Skills

These run the EPS → SVG conversion pipeline. Use these whenever you are processing sign files.

eps-to-svg — Main conversion pipeline

What it does: Converts signage .eps files into Brandography-ready .svg files for the Shopify live preview. Produces the full 4 or 5 file deliverable set (vector SVG, preview-vector, raster PNG, fields.json, FIELD_ANCHORS_Review.html) plus an optional .ai production file.

When Claude loads it: Any time you ask to convert EPS files to SVG, process a signage folder, prepare files for Brandography, or use phrases like "process the rest of the folder", "convert these signs", "run this on all the files", or "make SVGs from the EPS files."

What it encodes: All pipeline steps (1 → 13, plus 1d clip-mask and 9b .ai generation); structure-preserving XML tree walk approach; sign category classification (FOH/BOH/COA/illuminated/decal); face detection nesting method; bbox formula (cap_height + 1.5pt); anchor point requirements; font embedding; REVIEW SVG rules; self-audit + brand-delivery checks; .ai production file via Inkscape CLI (preferred), Illustrator JSX, or cairosvg PDF fallback.

Output: 5 files for variable signs, 4 for static. See the Outputs section of this guide for the full spec.

ℹ️ Script-vs-standard note — the current convert script is Motel 6 / Studio 6 specific. The general multi-brand standard documented in the skill is the source of truth; script gaps (face detection, static signs, multi-face EPS, PNG rendering, output filenames, fields.json schema) are being resolved as each brand is run.
⚠️ This skill works in Cowork only — it needs file system access to read EPS files and write SVG outputs. Running it in Chat will produce code that cannot execute.
clip-mask — AmericInn manufacturing junk preprocessor

What it does: A preprocessing step that runs before face detection on AmericInn signs. Finds the sign's border rectangle in the raw SVG and removes every element whose geometry falls outside it — profile views, dimension callouts, title blocks, manufacturing annotations. Geometry-based, not color-based, so it catches noise that color filtering misses.

When Claude loads it: Any AmericInn batch processing, or when you say "clip the signs", "clean up the SVGs", "preprocess before converting", "strip the junk", or "the pipeline is misclassifying manufacturing elements."

Runs: After EPS → PDF → raw SVG, before eps-to-svg face detection. The output is still a raw SVG — just cleaner input for the next step.

Why geometry beats color: Manufacturing junk (title block text, dimension callouts) shares the same near-black color as sign content (braille dots, sign text). Color filtering can't distinguish them. The clip mask uses position instead — if it's inside the sign border, it stays.

stairwell-informational — Universal stairwell sign layout spec

What it does: Codifies the universal 2-face stairwell informational sign pattern that is identical across all hotel brands — Intown Suites, AmericInn, Hyatt Place/Studios, Uptown Suites, and others. Face 1 = Exit Discharge (★ star). Face 2 = Standard directional (↓/↑ arrow). Each face is a separate SKU.

When Claude loads it: Any stairwell informational sign processing, "STAIR A sign", "exit discharge sign", "floor designation sign", "DOWN TO LEVEL 1", "UP TO LEVEL 1", or an EPS pairing a ★ face with a ↓ face.

Key rules it enforces:

  • 4 variable fields universal to both faces: stair_id, no_prefix, floor_designation, floor_range
  • 2 additional fields on the directional face: down_arrow (rotatable 0°/180°) and direction_word ("DOWN"/"UP") — these are always paired
  • "ROOF ACCESS" is static SVG text; only "NO," is the variable prefix
  • Arrow anchor_point = geometric center of arrow path = rotation pivot
  • direction_word bbox sized for "DOWN" (the longer option); "UP" left-aligns inside

📖Glossary

FOHFront of HouseSigns with the decorative backer (wood grain / gold strip). ViewBox must include the backer below the face.
BOHBack of HouseSigns without a backer — functional, utility signs. ViewBox is tight to the face only.
Variable sign—A sign with at least one live text field (fields[] non-empty). Room numbers, custom copy, directional destinations.
Static sign—Sign with no variable text (fields[] empty). Art is preserved exactly as-is from the EPS.
Magenta box—Dashed magenta rectangle in the EPS marking where a variable text field goes. Stripped from all output files — never delivered to the developer.
BboxBounding boxRectangle in SVG coordinate space defining where a live text field renders. Height = cap_height + 1.5pt. Bottom = text baseline.
Anchor point—Fixed (x, y) coordinate that the renderer holds pinned as text content changes. Required on every text and braille field.
anchor_h—Horizontal anchor mode: left, center, or right. Controls which edge of the content is held fixed.
anchor_v—Vertical anchor mode: always baseline for text and braille fields.
MSDQ—Master Sign Design Qualification — the manufacturer's PDF spec document. Source of physical dimensions, sign type codes, fonts.
E4 Workbook—Elaine's Excel workbook where all SKU numbers are assigned and tracked. Never modify without Elaine's authorization.
SKUStock Keeping UnitThe unique product number assigned to each sign variant in the workbook. Becomes the filename prefix for all output files.
pdftocairo—Poppler tool that converts PDF → raw SVG. Produces the structured SVG tree that the pipeline parses.
cap_height—Height of uppercase letters from baseline to top. Taken from the font's OS/2 table (sCapHeight). Never use font_size for bbox height.
braille_spacinggap_ptBaseline-to-baseline distance from the last text line to the braille line. ADA/fire inspection requirement — must appear in every braille field's fields.json.
RMNRoom NumberSign type code for guest room number signs. Variable text, usually 2 fields (room number + braille).
DIRDirectionalWayfinding sign with arrow(s) and destination labels. High field count (10+ normal). Arrow fields rotate; destination fields are live text.
pts_per_inch—Scale of the raw SVG coordinates. Standard = 27pt/in. Two-face side-by-side EPS = 18pt/in. Always verify before measuring.
FIELD_ANCHORS—Group of <rect> elements injected at the end of the SVG defining where each live field sits. Invisible in production, red in preview-vector.
Brandography—The Shopify developer who builds the live preview UI that uses the SVG files this pipeline produces.

💡Other Tips for Training

Before Your First Session

  • Open a few EPS files in Illustrator first. Find the magenta dashed boxes — these are your variable fields. Count them. This is the ground truth you verify Claude against.
  • Read the brand sign dictionary SKILL.md for the brand you're processing. It tells you exactly what to expect for each SKU before Claude starts.
  • Understand FOH vs BOH — you need to know which type you're looking at when reviewing SVG output. FOH = backer visible below the face. BOH = no backer.
  • Know the difference between static and variable. Magenta dashed boxes in the EPS = variable. No dashes = static. This determines which output files are required.

During a Session

Verify face selection first

When Claude processes a new sign type for the first time, check the raster PNG before reviewing the SVG. If it shows the assembly or side view instead of the face-on view, correct face detection before going further.

Trust the REVIEW SVG

The _REVIEW.svg with red overlay boxes is your fastest visual check. Open it in Chrome or Firefox. Red boxes should land exactly on the text — not above it, not extending into the sign edge.

Check the HTML review file

Open _FIELD_ANCHORS_Review.html in Chrome for every variable sign. Confirm the spec cards show correct font names, sizes, and bbox coordinates. Share with Elaine for final sign-off.

Never approve without the 3-point check

Before approving any sign: (1) braille spacing 1:1 to EPS, (2) SVG at 1:1 physical scale, (3) every field has anchor_point. All three must pass — no exceptions.

Giving Claude Corrections

Claude improves based on specific, precise feedback. Vague corrections ("that's wrong") take longer to fix than specific ones.

❌ Vague
  • "The boxes look wrong"
  • "Fix the font"
  • "The alignment is off"
  • "Try again"
✅ Precise
  • "The bbox_h for SKU 88622 is too tall — use cap_height + 1.5pt, not font_size"
  • "The font_family on room_braille should be FuturaBT-MediumCondensed, not AvenirLTStd-Medium"
  • "The anchor_point.y is 3pt below the baseline — re-read the baseline from the raw SVG use y value"
  • "Run Step 13 audit and show me the output"

April 13, 2026

✅ Three-point approval criteria ratified

A sign is auto-approved when all three conditions pass. Ratified at Uptown Suites SKU 88624.

  1. Braille spacing 1:1 to EPS — braille_spacing.gap_pt matches the measured baseline-to-baseline distance in the source EPS exactly.
  2. SVG at 1:1 physical scale — root SVG width and height in inches match the physical sign dimensions from the MSDQ.
  3. Anchor point on every field — every text and braille field has anchor_point, anchor_h, and anchor_v.

All three must pass. A sign that passes only two is not approved.

🔄 Uptown Suites — directional arrows are static print (not rotatable)

Uptown Suites only: Directional arrows are reverse flatbed digital print — they are NOT rotatable variable fields. No arrow fields appear in fields.json for SKUs 88627–88632. This is the opposite of Intown Suites behavior — do not apply one brand's rule to the other.

April 10, 2026

🐛 Bounding box baseline bug fixed — never use pdfplumber for baselines

Bug: Bounding boxes were not aligned to the bottom of the text — there was a gap below each box. Root cause: baselines were read from pdfplumber's char["bottom"], which sits ~0.282 × font_size below the true baseline.

Fix: Always read baselines from raw SVG <use y> values. pdfplumber's bottom must never be used as a baseline. Verified on 93493 / 88748.

Impact: Any fields.json generated before this fix with pdfplumber baselines will have bboxes shifted down. Regenerate if still in use.

🐛 ET namespace bug — ElementTree renames xlink and breaks text rendering

Bug: Preview-vector and REVIEW SVGs rendered with completely blank text. Root cause: ElementTree renamed xlink to ns1: when writing the SVG, breaking all <use xlink:href> references.

Fix: All SVG modifications must use string insertion, not ElementTree round-trips. Every SVG write now includes a smoke test: assert 'xlink:href' in output. If the assert fires, the file cannot be delivered.

🐛 Defs walker bug — pruning deleted glyph symbol paths

Bug: After pruning manufacturing junk from a preview SVG, all text rendered blank. Root cause: the prune walker descended into <defs> and deleted glyph <symbol> paths because they appeared as "paths outside the face bounds."

Fix: Prune walkers must skip <defs> entirely. After pruning, assert no empty symbols remain: assert not [s for s in root.iter('symbol') if not list(s)].

🐛 Raster magenta cleanup — dilated mask + face color required

Bug: Magenta dash ghosts remained visible in PNG previews, and replaced regions appeared as brighter-white patches against the sign face background.

Fix: Use a dilated mask (5px MaxFilter = ~2px dilation) to catch anti-aliased dash edges. Fill with the brand face background color, never [255,255,255] (pure white). Face background RGB values are in each brand's technical reference file.

Glyph groups (which are SVG-space positioned) still apply the geometry filter even in no-filter mode, to catch annotation callouts without hardcoding indices.
